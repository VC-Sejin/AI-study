# H.Point 앱 리뉴얼 구축 — RFP 분석 및 작업 로그

문서 출처: 현대백화점 H.Point 앱 리뉴얼 구축 제안요청서(RFP)
정리일: 2026-06-10

---

## 1. 사업 개요

- 사업명: H.Point 앱 리뉴얼 구축
- 발주기관: 현대백화점
- 사업기간(예상): 업체 선정 후 10개월 (구축 9개월 + 안정화 1개월)
- 선정방식: 제안 평가 결과 및 입찰 금액 종합
- 제안서 제출기한: 2026년 1월 23일

핵심 방향은 사용자 중심의 직관적 UX 개편, 집객 콘텐츠·고객 혜택 강화, 핀테크(충전 결제·리워드) 고도화, 검색·개인화 추천 도입, 앱·홈페이지 UI/UX 최신화, 보안 강화입니다.

## 2. 사업 대상 및 범위

리뉴얼 구축 영역은 UI/UX 전면 개편, 개인 맞춤 추천 콘텐츠, 광고 마케팅 고도화, 선물하기 고도화, 기부 고도화, 가입절차 간소화, 글로벌 기능 개선, 콘텐츠 고도화, 보안 강화입니다.

신규 서비스 구축 영역은 그룹사 올인원 허브, AI 챗봇(연동), 오픈형 유료 서베이, 금융 상품 중개, 포인트 자동충전 결제, 구독 채널 서비스입니다.

## 4. 기능 요구사항(SFR) 분류 — IA 직접 재료

기능 요구사항 69건은 정보구조(IA)와 화면설계의 직접 재료가 됩니다. 주제별로 묶으면 다음과 같습니다.

- 준비/제안: 수행 실적·모바일 시안(투데이·혜택 2본) 제출
- UI/UX 공통: 신규 BI 기반 기획·디자인, 웹/앱 접근성, 웹 반응형 리뉴얼, GA4·그루비·브랜치 태깅, 공유 최적화(OG·딥링크), 앱 버전 팝업, WAS 설계·품질
- 회원/보안: 가입·로그인 간소화, 앱투앱 SSO, 계열사 전환가입, 다이나믹 바코드(OTC), DPoP 인증
- 전시/광고/개인화/검색: 룰베이스 메인화면 동적 노출, 이벤트 페이지 모듈화, 그룹사 혜택 노출, 챗봇 연계 API, 메뉴 통합검색, LLM 챗봇, 커머스 캠페인 연동, 동영상 광고, 서브 메인(혜택·월렛·마이페이지), 사이드메뉴 동적 관리
- 캠페인: 빙고, 운영 고도화, 랜덤박스
- 콘텐츠/서비스: 오픈형 설문 연동, 자체 설문 업그레이드, 포인트워크 고도화, 퀴즈 리뉴얼, 글로벌 서비스 확대, 자동충전결제 월렛(선개발), 금융 상품 중개, 선물하기 고도화, 채널 구독, 기부 업그레이드, 하이라이트, Map 기반 사용처 안내
- 관리(Admin): 메인화면·금융중개·월렛·사용처·Live·캠페인·이벤트·글로벌·선물하기·버전·사이드메뉴·그룹사혜택·검색·광고·고객정보·설문·포인트워크·퀴즈·기부·하이라이트·쿠폰 관리
- 외부 어드민: 광고주 전용 외부 홈페이지(실적 대시보드)
- 데이터 수집/연동: 결제문자 수집, 계열사 혜택 I/F, 그루비 송수신, DW 적재, 데이터 이관, Branch 수신, 설문 연동

## 5. 비기능 요구사항 — 전 화면·시스템 공통 기준

- 성능(PER): 응답 1초 이내, 동시 로그인 55,000명, 전체 9백만명, 초당 100건 처리, 자원 사용율 기준 등
- 테스트(TER): 단위·종합·부하 테스트 계획·방안, 개발 착수 시 단위 테스트 병행, 발주사 TMS 활용
- 보안(SER, 필수): 보안정책·법규 준수, 단계별 보안관리, 내부망 보안, 보안검토 수검, 접속로그, 개인정보 암호화(SSL·TDE), 전 영역 모의해킹
- 품질(QUR): 위험관리, 품질보증 계획, 장애처리 방안
- 데이터(DAR): 발주사 데이터 모델링 기준(DA#·Meta#) 준수

## 6. 사업관리 성격 항목 — IA와 무관, 별도 분류

시스템 구성(ECR), 사업 추진(BPR), 수행 조직(POR), 교육(EDR), 대금(PAT)은 계약·관리 성격으로 IA·화면설계로 직접 이어지지 않습니다. 다만 추적을 위해 요구사항정의서에 누락 없이 포함했습니다.

주요 기술 표준으로는 Android(Kotlin 2.2.21, Compose, 클린아키텍처+Hilt), iOS(SwiftUI 기반 MVVM), OS 최소 지원(Android 9.0↑, iOS 16.0↑), Web(React/Vue/JSP 중 최적 제시, 기존 JSP), Native 35% 이상+Hybrid 구성, 금융망 서버 대역 분리(`26.04~11 일정 대응) 등이 명시되어 있습니다.

## 7. 진행한 작업

1. RFP(PDF) 전문을 검토하고 136건 요구사항을 식별했습니다.
2. 이를 요구사항정의서(엑셀)로 구조화했습니다. 1차 버전은 11개 칼럼(요구사항ID·RFP ID·구분·대분류·중분류·요구사항명·상세설명·신규/개선·적용범위·우선순위·관련 사용자)으로 작성했습니다.
3. 이후 2단 헤더 양식으로 재구성했습니다. 칼럼 그룹은 No / 요청 정보 / 요구사항 / 협의내용(1차) / 협의내용(2차) / 고객사 확인 순서입니다.
   - 요청 정보: 요구사항 ID, RFP ID, 요청단계, 요청경로, 요청일자, 요청자
   - 요구사항: 유형, 대분류, 중분류, 소분류, 담당자, 요구사항명, RFP요건, 추가내용, 신규/개선, 적용 범위
   - 협의내용(1차/2차): 구분, 제약사항, 해결방안 및 대안
   - 고객사 확인: 수용여부, 확인일자, IT검토자, 현업검토자
4. 작성용 빈칸 칼럼(요청단계·경로·일자·자, 소분류, 담당자, 추가내용, 협의내용, 고객사 확인)은 비워두었습니다.

## 8. 판단·가정 사항 (확정 필요)

- 우선순위(필수/중요/일반)는 RFP의 (필) 표시와 "선개발 필요"(월렛 자동충전, `26.8월 선오픈) 단서를 근거로 한 가안입니다. 발주사·내부 협의로 확정이 필요합니다.
- 적용범위(앱/웹/어드민/서버 등)는 상세설명에서 추론한 분류로, 조정될 수 있습니다.
- 협의내용 그룹은 RFP 요청에 따라 1차/2차 두 그룹으로 분리했습니다.
- 2단 헤더 재구성 시 기존 '우선순위'·'관련 사용자' 칼럼은 새 양식에 칸이 없어 제외했습니다(1차 버전 원본에는 보존).

## 9. 다음 단계 (예정)

기능 요구사항(FR) 69건에서 사용자 대면 메뉴를 도출해 IA(정보구조도)를 작성합니다. RFP의 모바일 시안 가이드(투데이/혜택 하단 탭)와 서브 메인 정의(혜택·월렛·마이페이지)가 메인 네비게이션의 핵심 단서입니다. 산출물 형태(트리형/화면ID 표/다이어그램)와 대상(앱 우선 / 앱·웹 동시)은 확정 예정입니다.
