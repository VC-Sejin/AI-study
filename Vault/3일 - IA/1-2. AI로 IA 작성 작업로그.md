# IA 작성 작업로그 — 현대백화점 NEW 커머스 플랫폼(더현대 Hi)

작성 기준 자료: 요구사항정의서(`2-1` / FO UI·UX, V0.8 시트) + 제안요청서(RFP, `1-1`)
산출물: `IA_더현대Hi.xlsx`

---

## 1. 개요

요구사항정의서와 제안요청서(RFP)를 기반으로 정보구조도(IA)를 작성했다.
초기 버전부터 단계적으로 ① 컬럼 구성, ② 개발 항목 제외, ③ 표준 커머스 앱 구조 적용,
④ 표준 하위 메뉴 세분화, ⑤ 참고 IA 양식(계단식) 적용, ⑥ 비고 항목 분리·출처 표기까지
총 6차에 걸쳐 수정·고도화했다.

---

## 2. 입력 자료 분석

### 2.1 제안요청서(RFP)
- 시트: `제안요청서_원본`
- 요구사항 분류 체계와 ID 규칙 확인:
  - 공통(RCR), 시스템구성(ECR), 기능(SFR, 176건), F/W(FWR), 성능(PER),
    테스트(TER), 보안(SER), 품질(QUR), 데이터(DAR), 사업추진(BPR) 등 총 259건.
  - 기능 요구사항(SFR)이 전시/검색, 상품, 주문, 물류, 회원, 상담원, 캠페인/프로모션/쿠폰,
    점기반 서비스, 연동 대응개발, 정산/회계, 조회/분석 등으로 세분화.
- RFP 본문을 파싱하여 **고유번호(SFR-xxx 등) → 요건 상세 설명** 딕셔너리 구성(227개 ID).

### 2.2 요구사항정의서
- 다수의 버전 시트 중 최신본 **`요구사항 정의서 (V0.8)`** 사용.
- FO UI/UX(바이널씨) 영역 정의서로, 464개 데이터 행.
- 주요 컬럼: NO, 요구사항ID, 담당모듈, 소분류, RFP 고유번호, 요구사항명, 세부내용 등.
- 병합 셀(NO·요구사항ID가 그룹 첫 행에만 존재)을 forward-fill 처리.
- 담당모듈 분포: 주문(144) · 전시(88) · 검색(55) · 회원(41) · 상품(34) · 이벤트(23) ·
  공통(16) · 프로모션(10) · 쿠폰(6) 등.
- 정의서의 RFP 고유번호와 RFP 본문 매핑 검증: 447/464행이 매칭(미매칭 3건은 RFP에
  본문이 없는 범위 밖 번호).

## 3. 단계별 작업 내역

### 3.1 [1차] 초기 IA 작성
- 요청 컬럼 구성: 번호 · 파트구분 · 큰 메뉴구조명 + 8뎁스 · 화면ID(생성) ·
  화면설명 · 요구사항ID · 요구사항ID 근거 · 화면구분.
- 담당모듈 → 파트구분 정규화, 소분류·요구사항명으로 메뉴 뎁스 구성.
- 화면ID 신규 생성(파트 코드 + 일련번호), 명칭 키워드로 화면구분 분류.
- 요구사항ID에 매칭되는 **RFP 본문 상세를 같은 행 근거 칸에 부착**.
- 결과: 464행.

### 3.2 [2차] 개발 항목 제외 + 뎁스 중복 제거 (이후 철회)
- 유형이 '개발'인 65건(검색엔진 색인, DB 작업 등 비화면 항목) 제외 → 399행.
- 상위 마디가 하위에 포함되면 상위 생략하는 뎁스 중복 제거 적용.
- ※ 이 차수의 "뎁스 중복 제거"는 다음 요청에서 **철회**됨.

### 3.3 [3차] 표준 커머스 앱 구조 적용
- 개발 항목 제외(유지)하고, 표준 커머스 IA 골격을 상위 구조로 재배치.
- 표준 IA 참고(홈/카테고리/검색/장바구니/체크아웃/마이페이지 등 핵심 섹션) 확인 후 적용.
- 큰 메뉴를 **FO(고객 서비스) / BO(운영 어드민)** 2영역으로 구분.
- 그 아래 표준 1뎁스 메뉴(홈/메인, 카테고리/전시, 검색, 상품상세, 장바구니,
  주문/결제, 마이페이지, 회원/인증, 혜택, 연동 등) 배치.
- 결과: FO 297행 / BO 102행 = 399행.

### 3.4 [4차] 표준 하위 메뉴 세분화
- 한 메뉴(특히 주문/결제 86건)에 항목이 몰리는 문제 해결.
- 키워드 기반으로 2뎁스 세부 메뉴 분리:
  - 주문/결제 → 주문서 · 결제/PG · 취소/교환/반품 · 배송/물류 · O4O·매장 주문 ·
    무형상품 주문 · 선물하기
  - 카테고리/전시 → 브랜드관 · 전문관/테마관 · 카테고리 매장 · 기획전 ·
    광고/배너 구좌 · 페이지 빌더 · 콘텐츠/라이브 · 특화관
  - 검색 → 검색 영역 · 검색 결과 · 필터/정렬 · 검색어/추천어 · 검색 수단
  - 회원/인증 → 로그인/인증 · 회원가입/통합 · 등급 · 포인트/적립금 · 멤버십 · 정보/설정
  - BO도 동일 방식으로 관리 메뉴 세분(배송/물류·취소반품CS·정산매출 등).
- 복합 모듈(예: "주문\n상품") 첫 토큰 추출 로직 보정으로 분류 정확도 향상.

### 3.5 [5차] 참고 IA 양식(계단식) 적용
- 참고 파일 2종 분석:
  - **삼성카드쇼핑 IA정의서**: `구분 | 1depth ~ 7depth | 비고` 계단식 구조,
    화면구분을 메뉴명 접두사(T=탭, t=팝업)로 표기, 비고에 컴포넌트/설명 기재.
  - **삼성미술관 리움 Back IA**: `NO | 1depth ~ Ndepth | 비고`, 계단식 + 비고 설명.
- 사용자 결정:
  - 양식 그대로(계단식 depth) 적용, 요구사항ID·근거는 비고로 통합.
  - 화면구분은 별도 컬럼 유지(화면/팝업/프로세스).
- 적용 결과:
  - 컬럼: `NO | 구분 | 1~6depth | 비고`.
  - 1depth=영역, 2depth=표준메뉴, 3depth=세부메뉴, 4~6depth=정의서 하위 화면.
  - 상위 메뉴는 첫 행에만 표기하고 하위는 빈칸으로 내려가는 **계단식(cascading)** 표현.
  - depth별 음영 적용으로 위계 가독성 확보.

### 3.6 [6차] 비고 분리 + 임의생성/원본 출처 표기
- 비고 통합 칸을 개별 컬럼으로 분리:
  `화면 ID | 요구사항 ID | 요구사항 ID 근거 | 화면 설명 | 생성 구분`.
- 출처(provenance) 표기:
  - **임의생성(빨강)**: 화면ID, 구분, 1depth(영역), 2depth(표준메뉴), 3depth(세부메뉴).
  - **원본 인용(회색)**: 요구사항ID(RFP 고유번호), 근거(RFP 본문),
    화면설명(정의서 세부내용), 4~6depth(정의서 소분류·요구사항명).
  - **생성 구분 컬럼(녹색)**에 행별 명시:
    - `원본(RFP/정의서) + IA분류 임의생성` (383건)
    - `IA분류 임의생성 / 요구사항ID 미지정(정의서 메모)` (16건, 노란 음영)
  - 요구사항ID 없는 16건은 정의서 원본 메모("(추가사항)", "3.0에 아예 없음", "중복" 등)를
    근거 칸에 그대로 기재.
- 상단 범례로 색·음영 의미 명시.

## 5. 분류 기준 요약

### 5.1 영역 판정
- 어드민/관리자/등록·수정·승인/HM·HBO 코드 또는 유형='어드민' → **BO**
- 그 외 고객 대상 화면 → **FO**

### 5.2 화면구분
- 팝업/모달/알럿/레이어 키워드 → **팝업**
- API/연동/배치/스케줄 등 → **프로세스**
- 나머지 → **화면**

### 5.3 요구사항ID 근거
- 정의서의 RFP 고유번호를 키로 RFP 본문 요건 상세를 부착.
- 본문이 없는 번호는 "(RFP 본문에 상세 없음)", 고유번호 자체가 없으면 정의서 메모 기재.

## 6. 검토·확인 필요 사항 (Open Issues)

1. **요구사항ID 미지정 16건**: 정의서에 RFP 매핑이 비어 있는 항목(추가사항/3.0 미존재/중복).
   별도 요건 확정 또는 RFP 신규 항목 매핑 필요.
2. **정의서 범위**: 현재 IA는 FO UI/UX(바이널씨) 정의서 기준. BO·연동 전체를 포함한
   풀 IA가 필요하면 별도 정의서 시트/자료 확인 필요.
3. **양식 확장 옵션**: 영역별·표준 메뉴별 시트 분리, 각 depth 칸에 (생성)/(원본) 태그
   직접 부착 등 추가 정밀화 가능.
